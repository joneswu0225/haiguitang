# RESEARCH.md — AI 海龟汤游戏

## 1. 我们要做什么

开发一款 **AI 海龟汤** 产品：由 AI 担任「煲汤人」（主持人/裁判），玩家通过封闭式提问推理出完整真相；AI 仅回答「是 / 不是 / 无关（或不重要）」等有限口径，并在适当时机揭示汤底或判定胜负。

## 2. 为什么要做

- **随时可玩**：不依赖真人朋友组局，单人也能完整体验推理闭环。
- **题库与难度可扩展**：可结合生成或 curated 题库，覆盖入门到高难度。
- **一致规则执行**：AI 可严格维持「只能三答」等规则，减少真人主持口误或剧透。
- **学习成本低**：适合作为休闲推理与逻辑表达练习的载体。

## 3. 海龟汤（情境推理）完整规则（传统版）

### 3.1 名称与来源

「海龟汤」是国内对一类 **水平思考 / 情境猜谜**（lateral thinking puzzle）的俗称，名称来自早期流传的一道经典题目（男子喝海龟汤后自杀等变体）。玩法核心与是否使用「海龟」意象无关。

### 3.2 核心术语

| 术语 | 含义 |
|------|------|
| **汤面** | 出题者给出的不完整、常显反常或悬疑的事件描述（谜面）。 |
| **汤底** | 事件背后的完整真相与关键因果链（标准答案）。 |
| **煲汤人** | 掌握汤底、负责作答的一方（传统游戏中为真人）。 |
| **喝汤人** | 通过提问推理真相的一方（可多人协作）。 |

### 3.3 基本流程

1. 煲汤人公布 **汤面**（不公布汤底）。
2. 喝汤人轮流（或协作）提出 **问题**。
3. 煲汤人仅依据汤底判断每个问题，并给出 **有限集合的回答**（见下）。
4. 当喝汤人 **完整、准确地讲出汤底**（或达到约定等价条件）时，游戏结束；煲汤人可补充讲解或复盘。

### 3.4 提问与回答的硬性约束（最常见规则）

- 喝汤人应尽量提 **封闭式问题**（是否、有没有、是不是因为……），便于裁判判真假。
- 煲汤人通常 **只能** 回答以下之一（具体措辞可产品化统一）：
  - **是** — 与汤底一致。
  - **不是** — 与汤底矛盾。
  - **无关 / 不重要** — 无法从汤底判定，或对还原真相无关键影响（不同圈子对「无关」定义略有差异，产品内需写死规则）。
- 部分变体允许 **「是也不是」**（半对半错）或 **「问题无效」**（非封闭、依赖假设过多）— 若做进产品，须在规则页明确是否启用。

### 3.5 胜负与体验约定（常见）

- **胜利条件**：口述还原汤底的关键事实链（产品可定义：关键词覆盖、结构化提交、或 AI 判定「等价于汤底」）。
- **失败/放弃**：主动查看汤底、或超时/次数限制用尽（若产品设计中有）。

## 4. AI 版海龟汤 vs 传统版：有什么不同？

| 维度 | 传统版 | AI 版 |
|------|--------|--------|
| **煲汤人** | 真人，灵活、有情绪与互动 | 模型 + 规则层，需防止胡答、剧透、人设漂移 |
| **汤底一致性** | 依赖真人记忆，偶有矛盾 | 应以结构化汤底为唯一真源，回答可校验 |
| **三答口径** | 靠自觉，易有争议 | 可由提示词 + 校验逻辑强制执行 |
| **社交** | 强社交、线下/语音氛围 | 偏单人或异步；可后续加房间、观战、协作 |
| **题库** | 自带题目或即兴编 | 可静态题库、可生成、需 **安全与合规** 审核（暴力、敏感题材） |
| **解释与复盘** | 真人自然讲解 | AI 可在结束后生成复盘，但要避免与对局中的「无关」回答自相矛盾 |
| **成本** | 人力时间 | 推理 token、延迟、API 费用 |

**产品启示**：AI 版的核心工程问题不是「会不会聊天」，而是 **(1) 汤底结构化**、**(2) 每轮问答与汤底的可判定性**、**(3) 不剧透**、**(4) 失败时的降级与纠错**。

## 5. 是否已有类似产品（简况）

- 线上海龟汤社区、题库站、小程序等：多为 **真人主持** 或 **固定选项**，与「开放式提问 + 三答」的 AI 裁判组合不同。
- 「AI 猜谜 / AI DM」类产品：形态接近，可借鉴其 **系统提示词结构、状态机、安全策略**，但需对齐海龟汤的 **严格三答** 与 **汤底保密**。

（具体竞品与差异化可在 PRD 中展开；此处仅作研究阶段占位。）

## 6. 建议的必备核心功能（MVP 导向）

以下为 **AI 海龟汤** 若要「可玩且像海龟汤」的必备项；优先级与裁剪在 [PRD.md](./PRD.md) 中细化。

1. **单局游戏闭环**  
   选题 → 展示汤面 → 多轮问答 → 胜利判定或公布汤底 → 简短复盘。

2. **严格裁判模式**  
   对用户每一问输出 **是 / 不是 / 无关**（或产品统一口径），且 **不主动泄露** 汤底关键信息。

3. **结构化汤底（机器可读）**  
   后台保存完整真相、关键事实点、禁止直接透露的词或情节；供模型或规则层约束回答。（可与「仅提示词里塞一段故事」的简陋方案区分，后者难维护。）

4. **汤面与难度**  
   至少支持题库中选題（或预置若干题）；可选标注难度、标签（恐怖/推理/搞笑等）与 **内容安全分级**。

5. **局内状态**  
   当前轮次、可选的提问历史展示，避免用户重复劳动；可选「请求提示」作为弱辅助（若做，需定义是否破坏硬核体验）。

6. **基础合规与安全**  
   用户生成内容（若开放投稿）与 AI 生成内容的审核策略；拒绝现实伤害教程、过度血腥等（细则随上架平台调整）。

7. **失败与边界**  
   问题无法理解时的处理（请用户改写成是否问句）；模型异常时的重试或降级文案。

## 7. 研究阶段待决问题（交给 PRD / 技术设计）

- 「无关」的判定标准：严格按汤底可判定性，还是引入「对推理帮助程度」？
- 胜利条件：必须逐字还原，还是 AI 判定语义等价即可？
- 是否允许用户 **自定义汤底** 让 AI 主持他人来玩（UGC）？
- 首发形态：Web / 小程序 / App；是否需要账号与存档？
